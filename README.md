# Скиллы

Переиспользуемые скиллы для AI-агентов от Кирилла BuhaiStrikalo.

В этом репозитории я храню, улучшаю и публикую небольшие практичные скиллы, которые помогают Codex и другим совместимым AI-агентам лучше выполнять конкретные повторяющиеся задачи.

Идея простая: хороший скилл - это не длинный промпт. Это компактная рабочая процедура для задачи, где важен опыт: что проверить первым делом, каких ошибок избегать, какой результат выдать и как убедиться, что работа действительно сделана.

## Установка

```bash
npx skills@latest add buhaistrikalo/skills
```

После установки перезапустите Codex, чтобы новые скиллы появились в доступном списке.

## Доступные скиллы

| Скилл | Что делает |
| --- | --- |
| [`figma-fix`](./.agents/skills/figma-fix/SKILL.md) | Помогает внести точечное frontend-изменение по Figma-ноде, скриншоту или визуальному референсу. |
| [`handoff`](./.agents/skills/handoff/SKILL.md) | Помогает подготовить короткий handoff-документ для следующего агента или новой сессии. |
| [`push-to-main`](./.agents/skills/push-to-main/SKILL.md) | Помогает аккуратно закоммитить и запушить изменения в `main`, не захватив лишние файлы. |
| [`skill-maker`](./.agents/skills/skill-maker/SKILL.md) | Помогает превратить узкий workflow или экспертизу в короткий переиспользуемый скилл для агента. |
| [`small-diff`](./.agents/skills/small-diff/SKILL.md) | Помогает уменьшить разросшийся diff до действительно нужных изменений. |
| [`trace-bug`](./.agents/skills/trace-bug/SKILL.md) | Помогает объяснить баг через реальный путь кода и данных, а не через догадки. |

## Структура репозитория

```text
.agents/
  skills/
    <имя-скилла>/
      SKILL.md
      agents/
        openai.yaml
      scripts/
      references/
      assets/
```

Обязателен только `SKILL.md`. Остальные папки опциональны:

- `agents/openai.yaml` добавляет UI-метаданные для списка скиллов в Codex.
- `scripts/` подходит для повторяемых команд или хрупких преобразований.
- `references/` подходит для длинных документов, которые агент должен читать только при необходимости.
- `assets/` подходит для шаблонов, изображений, boilerplate-кода и других файлов, которые используются в итоговых артефактах.

## Что делает скилл хорошим

Хорошие скиллы узкие, практичные и имеют понятную точку зрения.

Лучше:

- "Починить упавший GitHub Actions по логам"
- "Написать страницу альтернативы конкуренту"
- "Проверить события продуктовой аналитики"
- "Составить founder-led launch plan"
- "Уменьшить раздутый diff"
- "Запушить изменения в main без лишних файлов"

Хуже:

- "Маркетинг"
- "DevOps"
- "Письмо"
- "Продуктивность"

Полезный скилл обычно содержит:

- конкретное описание триггера
- первый шаг, который должен сделать агент
- ключевые развилки и решения
- типичные ошибки
- ожидаемый формат результата
- шаги проверки

## Как создать новый скилл

Создайте новую папку внутри `.agents/skills/`:

```text
.agents/skills/my-skill/
  SKILL.md
  agents/openai.yaml
```

Минимальный `SKILL.md`:

```markdown
---
name: my-skill
description: Использовать, когда пользователь хочет ...
metadata:
  short-description: Делает одну полезную вещь
---

# My Skill

Одна фраза о том, в какой задаче этот скилл делает агента сильнее.

## Workflow

1. Начать с реального источника правды.
2. Выполнить процедуру для конкретной задачи.
3. Проверить результат перед финальным ответом.
```

Минимальный `agents/openai.yaml`:

```yaml
display_name: My Skill
short_description: Делает одну полезную вещь
default_prompt: Помоги мне применить этот workflow.
```

## Контрибьютинг

Контрибьюты приветствуются, если они помогают агентам лучше делать настоящую работу.

Перед pull request:

- Держите скилл сфокусированным на одной задаче.
- Пишите инструкции достаточно коротко, чтобы их было комфортно загружать в контекст.
- Длинные примеры и справочные материалы кладите в `references/`.
- Добавляйте скрипты только тогда, когда они убирают хрупкую повторяемую работу.
- Проверьте, что описание триггера достаточно ясное и агент поймет, когда использовать скилл.

## Философия

Скиллы - это упаковка опыта и суждения.

Лучшие скиллы делают экспертизу видимой: показывают, как опытный человек подходит к задаче, куда смотрит сначала и что отказывается замазывать общими словами.
